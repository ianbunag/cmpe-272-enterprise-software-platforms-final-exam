# AI usage writeup

Claude served as the primary implementation engine for both secure transfer protocols, authoring the majority of the functional logic once the core specifications were defined.

In Approach A, the model generated the end-to-end mutual TLS 1.3 listener and the binary streaming logic, while in Approach B, it developed the hybrid encryption logic including the RSA-OAEP envelope and the AES-GCM chunking mechanism. Architectural boundaries and security constraints were strictly dictated by Gemini Plus, which acted as the lead architect by processing the original examination documents and assessment checklists to evaluate the trade-offs between transport-layer and application-layer security.

The high-quality task requirements and checklists provided at the start prevented the AI from proposing inherently insecure cryptographic designs, though technical friction occurred during the implementation phase. A concrete example was found in Approach B, where Claude initially proposed a session resumption strategy that only checked for the existence of a file to determine the starting offset. I identified this as a logic flaw that could allow a malicious sender to overwrite existing data. I caught this by reviewing the state management logic and directed Claude to implement a more robust handshake where the receiver verifies a unique session ID and a signed manifest before allowing bytes to be appended.

Claude performed better than expected in its idiomatic use of Go’s standard library, particularly with the io.MultiWriter and io.CopyBuffer interfaces, which allowed for a highly efficient "pass-through" architecture where data is written to disk and hashed for integrity simultaneously. Conversely, the model performed worse than expected regarding code hygiene and service stability. It frequently left unused variables and redundant helper functions in the source code, and its handling of TCP frame boundaries was initially fragile. I had to manually intervene to ensure the receiver could handle abrupt connection drops without hanging or leaving partial files.

The development workflow utilized a three-tier AI stack to ensure a defensible final product. Gemini Plus functioned as the strategic planner, providing the initial specification documents and staged implementation prompts. Claude then executed those prompts to build the applications and generate the necessary documentation. Throughout this process, GitHub Copilot served as a real-time peer reviewer within the IDE, providing quick fixes for syntax errors and generating the multi-stage Dockerfiles needed for the simulation environment. This collaboration allowed for a robust implementation that satisfies the Confidentiality, Integrity, Authenticity, and Availability requirements of the exam.
