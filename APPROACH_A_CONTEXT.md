## The runtime/ Directory Pattern

Every file that is generated at runtime goes inside `runtime/`. Nothing generated is ever
committed. Each subdirectory contains a `.gitkeep` file so the directory exists on a fresh
clone without any user action. The `.gitignore` uses the `runtime/<dir>/*` pattern with
`!runtime/<dir>/.gitkeep` exceptions.

This pattern must be replicated for Approach B in its own subdirectory.

---

## Language and Build

Go was chosen for its standard library TLS 1.3 support, idiomatic streaming I/O, and static
binary output. All code uses only the Go standard library — no third-party packages.

Binaries are built with `CGO_ENABLED=0` inside the Docker builder stage so the final image
has no C runtime dependency and can run on Alpine with no extra packages.

**Important:** Go binaries built locally on the development machine (macOS 26) crash at
launch due to an OS compatibility issue. Do not attempt to run or verify binaries outside
of Docker. All testing and verification must happen inside the containers.

---

## Docker Pattern

Each approach has its own independent `docker-compose.yml`. Services are never auto-started
into the program. Instead:

- Each Dockerfile ends with `CMD ["sleep", "infinity"]`
- `stdin_open: true` and `tty: true` are set on every service
- The user runs `docker compose up -d` then `docker compose exec <service> sh` to enter
  each container and run the program manually

This makes the transfer interactive and lets the user observe logs in real time without
needing separate log tailing.

The three services are `sender`, `receiver`, and `attacker`. All share a private bridge
network named `transfer-net`. The Docker service name `receiver` doubles as the hostname
the sender dials and the DNS name in the receiver's TLS certificate SAN.

Volume mounts follow this convention:

| Host path              | Container path | Permissions |
|------------------------|----------------|-------------|
| runtime/certs/         | /certs         | read-only   |
| runtime/data/          | /data          | read-only   |
| runtime/output/        | /output        | read-write  |

## Certificate Architecture

Certificates are generated by `gen_certs.sh` using OpenSSL and written to `runtime/certs/`.
The script is idempotent in the sense that it overwrites any existing files. Users should
re-run it if they need fresh credentials.

Key decisions:

- Root CA: RSA 4096-bit, 10-year validity, self-signed
- Sender cert: RSA 4096-bit, 365-day, signed by Root CA, `extendedKeyUsage = clientAuth`
- Receiver cert: RSA 4096-bit, 365-day, signed by Root CA, `extendedKeyUsage = serverAuth`,
  `subjectAltName = DNS:receiver,IP:127.0.0.1`

The receiver SAN is required because Go 1.15+ rejects server certificates with no SAN
even if the CN matches. The IP:127.0.0.1 SAN covers local loopback testing if needed.

No certificate material is baked into any Docker image or stored in environment variables.
All certs are mounted as read-only volumes.

## The Application Protocol (Approach A Specific)

The sender and receiver share a simple two-part framing over the TLS stream:

1. A newline-terminated JSON header sent before any file data:
   `{"filename":"...","expected_sha256":"...","file_size":...}\n`

2. Raw binary file bytes with no additional framing until the connection closes.

The receiver uses `bufio.NewReaderSize` to read both parts through the same reader so no
bytes are lost at the header/data boundary.

Both programs use a fixed 32 KB buffer. The receiver writes chunks simultaneously to disk
and to a running SHA-256 hasher using `io.MultiWriter`.

## The Fail-Closed Pattern

The receiver never writes to the final filename directly. It always writes to a `.tmp` file
and uses a `defer` with a `success` boolean flag:

```go
success := false
defer func() {
    tmpFile.Close()
    if !success {
        os.Remove(tmpPath)
    }
}()
// ... transfer ...
// only set after hash verified:
success = true
os.Rename(tmpPath, finalPath)
```

Any error path — network drop, size mismatch, hash mismatch — returns without setting
`success`, which fires the cleanup. This pattern must be replicated in Approach B wherever
a file is written.

## Logging Conventions

Both sender and receiver log:

- A startup message confirming the address or file being processed
- A connection event when a peer connects or the sender connects to the receiver
- Progress every 100 MB including percentage and throughput in MB/s
- A final summary with total MB, elapsed time, and average throughput
- Specific error messages for every failure path

All log messages use `log.Printf`. No secrets (keys, hashes of keys, passphrases) are
ever printed to logs.

## Prose Constraints in Code and Documentation

All comments in source files and all prose in markdown files must avoid dashes, colons,
and semicolons within sentences. This was specified in the original plan. Code syntax
(flags, struct tags, shell commands) is exempt — only natural language sentences are
affected.

## What Approach B Must Do Differently

The assessment requires two approaches that differ in a meaningful architectural way, not
just by cipher suite. Approach A uses transport-layer security (mTLS) with an online
interactive key exchange. Approach B must differ in one of these dimensions:

- Application-layer envelope encryption instead of transport-layer TLS
- Pre-distributed long-lived keys (PSK or recipient public key) instead of online ECDHE
- Chunked-and-signed manifest with resumable parts instead of a single stream
- A sender-to-broker-to-receiver path where the broker is untrusted with plaintext

Whichever architecture is chosen, Approach B must still satisfy all CIAA properties and
address all six threat model rows. It must also have its own independent `docker-compose.yml`
and the same four documentation files listed above.
